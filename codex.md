# Codex 项目全局设定

## 1. 项目定位

- 本项目按正式上线运行的前端页面标准进行开发，不按产品原型稿方式输出。
- 本项目当前只考虑前端页面内容、交互、样式。
- 所有数据使用 mock 填充，不对接接口，不接数据库。

## 2. 开发原则

- 所有改动遵循最小代码改动原则。
- 只改动当前需求涉及的功能代码，其他代码不要动。
- 不自作主张做额外优化。
- 不以“便于后续维护”为由扩散修改范围。
- 一个页面对应一个代码文件，不把多个页面堆在同一个页面代码文件中实现。
- 所有页面改动除功能实现外，还要同步考虑 UI 美化，保证页面具备正式上线的视觉质量。

## 3. 页面输出标准

- 页面按正式交付页面处理，不展示原型说明型文案、内部备注型文案、演示提示型文案。
- 页面中所有按钮、入口、操作项点击后都要有对应效果，保证交互闭环。
- 页面中的排版、样式、信息层级应以正式上线观感为准，不使用“功能说明代替布局”。
- 页面设计需同步关注视觉美感，包括布局比例、留白、对齐、层级、色彩、组件质感和整体统一性。
- 图片是图片、合同是合同、地图是地图、字段是字段，不用一句概括式文案替代真实展示结构。
- 删除无意义的英文眉标题、解释性说明、副标题、标签说明、数字说明时，优先保留真实业务信息。

## 4. 需求执行流程

- 每次功能性改动前，先输出功能结构清单。
- 功能结构清单需明确到页面、弹窗、字段、必填项、按钮和交互效果。
- 按清单执行页面开发，不跳步骤。
- 每次代码修改后检查是否出现乱码，如有乱码必须立即修正。

## 5. 业务口径

### 5.1 自提

- `自提` 的含义是：个人自行取货。
- `自提` 不是寄送到服务商门店后，再去服务商门店提货。
- 后续所有页面、文案、交互中涉及 `自提` 时，统一按本口径处理。

### 5.2 服务商展示

- 服务商端不能决定平台侧才有权限控制的内容。
- 例如案例展示状态这类能力，由平台侧统一审核和处理，服务商端不提供该控制权。

## 6. 文案规范

- 不在正式页面中展示“mock 提示”“演示说明”“此处显示”“建议展示”“内部说明”等原型化文案。
- 不在客户可见页面中保留不必要的英文说明标题。
- 文案以业务表达为主，不使用开发者视角说明。

## 7. 当前协作约定

- 默认使用中文沟通。
- 修改完成后，需说明本次改动范围、落地文件和检查结果。
- 若需求与已有业务口径冲突，以本文件中的全局设定为准；如需调整，先更新本文件。
